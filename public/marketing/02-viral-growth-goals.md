# mdocks Viral Growth Goals

The goal is to make mdocks easy to understand, easy to try, and easy to share. For developer tools, "viral" usually means developers quickly see the value, trust the product, star it on GitHub, and share it with other developers who have the same problem.

## Main Growth Goal

Make mdocks known as:

> The private Markdown reader for local files and GitHub repo docs.

Everything should support this idea.

## Upcoming Product Goals

## 1. Make The First Impression Strong

Agenda:

- Add a clear product screenshot near the top of the homepage.
- Add a short demo GIF or video showing a local Markdown folder opening in mdocks.
- Make the main CTA obvious: `Open demo`, `Run locally`, and `View GitHub`.
- Add a short privacy line near the CTA: `Your docs stay on your machine.`

Success metric:

- More visitors click `Open demo` or copy the run command.

## 2. Improve The GitHub Repository

Agenda:

- Add a strong README headline.
- Add screenshots or a GIF.
- Add a fast start section.
- Add a "Why mdocks?" section.
- Add a privacy explanation.
- Add repository topics like `markdown`, `docs`, `local-first`, `developer-tools`, and `github`.
- Add a custom GitHub social preview image.

Success metric:

- More GitHub stars and more visitors coming from GitHub.

## 3. Create Shareable Demo Content

Agenda:

- Record a 30-second video showing the full flow from homepage to demo viewer.
- Record a 30-second video showing local folder browsing.
- Record a 30-second video showing local server GitHub repo browsing.
- Create 3 simple screenshots with short captions.

Success metric:

- These assets can be reused on Product Hunt, Hacker News, Reddit, X, LinkedIn, README, and blog posts.

## 4. Make The Demo Feel Useful

Agenda:

- Improve the bundled demo document so it explains mdocks while showing real Markdown rendering.
- Add sections in the demo for headings, code blocks, tables, links, frontmatter, and table of contents.
- Make sure `/viewer?demo=mdocs-info` feels polished and fast.

Success metric:

- Visitors understand the product without installing it.

## 5. Add SEO Pages

Agenda:

- Create `/local-markdown-reader`.
- Create `/private-markdown-viewer`.
- Create `/github-repo-docs-browser`.
- Create `/compare/obsidian`.
- Create `/compare/vscode-markdown-preview`.

Each page should include:

- A specific problem statement
- Why mdocks solves it
- Product screenshot
- Demo CTA
- GitHub CTA

Success metric:

- More organic search traffic over time.

## Community Launch Agenda

## 1. Soft Launch To Friendly Developers

Agenda:

- Share mdocks with 10 to 20 developers before public launch.
- Ask what is confusing about the homepage.
- Ask what would stop them from trying it.
- Ask whether the privacy/local-first message is clear.
- Fix the top 3 repeated objections.

Success metric:

- The homepage and README become clearer before public traffic arrives.

## 2. Launch On Hacker News

Agenda:

- Prepare a Show HN post.
- Use this title:

> Show HN: mdocks - a local-first Markdown reader and GitHub repo browser

- Keep the post honest and direct.
- Explain why it exists.
- Link to the demo, GitHub repo, and website.
- Stay active in comments for the full day.

Success metric:

- Comments, GitHub stars, useful feedback, and repeat visitors.

## 3. Launch On Reddit

Agenda:

- Post only in relevant communities.
- Customize the post for each community.
- Focus on feedback, not promotion.
- Reply thoughtfully to comments.

Potential communities:

- `r/selfhosted`
- `r/opensource`
- `r/webdev`
- `r/SideProject`
- `r/markdown`
- `r/github`

Post angle:

> I built a local-first Markdown reader because I did not want to upload private docs. Looking for feedback from developers who use Markdown folders or repo docs.

Success metric:

- Real conversations, feedback, stars, and demo opens.

## 4. Launch On Product Hunt

Agenda:

- Launch only after the demo, screenshots, README, and homepage are polished.
- Prepare a short demo video.
- Add 4 screenshots.
- Use this tagline:

> Local-first Markdown reader for developers.

- Write a first comment explaining the problem, privacy angle, and use cases.
- Ask early users for honest support.

Success metric:

- Product Hunt visits, backlinks, stars, and social proof.

## Virality Checklist

Before a major launch, make sure mdocks has:

- A clear one-line pitch
- A polished homepage
- A polished GitHub README
- A demo that works without setup
- A short demo video or GIF
- Screenshots that show the actual product
- A strong privacy message
- A simple run command
- A GitHub star CTA
- Shareable posts ready for each channel

## 30-Day Growth Agenda

## Week 1: Polish

- Improve homepage hero copy.
- Add product screenshot or GIF.
- Improve README.
- Add GitHub social preview.
- Improve bundled demo document.

## Week 2: Validate

- Ask 10 to 20 developers for feedback.
- Publish 1 or 2 SEO pages.
- Post build-in-public updates.
- Fix confusing onboarding points.

## Week 3: Developer Launch

- Launch on Hacker News.
- Share on selected Reddit communities.
- Post demo clips on X and LinkedIn.
- Collect feedback and ship improvements quickly.

## Week 4: Wider Launch

- Launch on Product Hunt.
- Publish a technical blog post.
- Share results and lessons learned.
- Add testimonials or quotes to the website.

## North Star Metric

Use GitHub stars and demo opens as the first growth signals.

Early goal:

- 100 GitHub stars
- 500 demo opens
- 20 useful feedback comments
- 5 public mentions or backlinks

Next goal:

- 500 GitHub stars
- 2,500 demo opens
- 100 command copy clicks
- 25 public mentions or backlinks

## Final Agenda

The priority is not to chase every channel. The priority is to make mdocks so clear that developers can describe it in one sentence:

> mdocks is a private Markdown reader for local files and GitHub repo docs.

Once that sentence is obvious from the website, README, screenshots, and demo, every launch channel will perform better.
