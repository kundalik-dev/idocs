# mdocks Marketing Plan

mdocks should be marketed as a private, local-first Markdown reader for developers. The strongest message is simple: open local Markdown files, folders, and GitHub repo docs in a clean reader without uploading anything.

## Positioning

Use this core positioning across the website, README, social posts, and launch pages:

> Read Markdown docs locally. Browse GitHub repos privately.

mdocks is for developers who want a focused way to read Markdown-heavy folders, private documentation, and repo docs without moving content into another hosted platform.

## Primary Website Goal

The website should get visitors to try mdocks immediately. The homepage should make the value clear in the first few seconds and push users toward one of three actions:

- Open the demo
- Run mdocks locally
- View the GitHub repository

Recommended homepage copy:

> mdocks lets developers open local Markdown files, folders, and server-backed GitHub repos in a clean reader. Your documents stay on your machine.

## Target Audiences

Start with developers and technical users who already work with Markdown.

- Developers with Markdown-heavy projects
- Open-source maintainers
- Technical writers
- Indie hackers
- Privacy-conscious developers
- Small teams with private internal docs

Avoid broad enterprise positioning at the beginning. mdocks is better suited to developer-led adoption first.

## Audience Messages

For individual developers:

> Open your Markdown folders in a clean reader without leaving the browser.

For open-source maintainers:

> Let contributors browse repo docs locally in a focused reader.

For small teams:

> Read private repo documentation without moving it into another hosted platform.

For privacy-focused users:

> Your docs stay local. No uploads, no account, no cloud workspace.

## Website Pages To Add

Add focused pages that match the problems people search for:

- `/local-markdown-reader`
- `/github-repo-docs-browser`
- `/private-markdown-viewer`
- `/compare/obsidian`
- `/compare/vscode-markdown-preview`
- `/docs`

Each page should include a clear CTA to open the demo, copy the run command, or visit GitHub.

## Best Channels

## GitHub

GitHub should be the main distribution engine.

Actions:

- Improve the README with screenshots, a short GIF, install command, demo link, and privacy explanation.
- Add repository topics such as `markdown`, `docs`, `documentation`, `local-first`, `nextjs`, `developer-tools`, and `github`.
- Add a custom social preview image.
- Add a visible "Star on GitHub" CTA on the website.
- Create beginner-friendly issues if the project is open source.

Goal:

Build trust, stars, and organic developer discovery.

## Hacker News

Launch with a simple and honest Show HN post.

Suggested title:

> Show HN: mdocks - a local-first Markdown reader and GitHub repo browser

Post angle:

- Markdown docs are often scattered across folders and repos.
- mdocks runs locally.
- Documents do not leave the machine.
- A demo is available for people who do not want to install yet.

Goal:

Get developer feedback, GitHub stars, and early adopters.

## Product Hunt

Use Product Hunt after the website, demo, README, and screenshots are polished.

Launch assets:

- Short demo video
- Four product screenshots
- Clear tagline: `Local-first Markdown reader for developers`
- First comment explaining privacy, local files, and GitHub repo browsing

Goal:

Generate social proof, backlinks, feedback, and awareness.

## Reddit

Use Reddit carefully. The best strategy is to be helpful and relevant, not promotional.

Potential communities:

- `r/selfhosted`
- `r/opensource`
- `r/webdev`
- `r/SideProject`
- `r/markdown`
- `r/github`

Good post angles:

- "I built a local-first Markdown reader because I did not want to upload private docs"
- "How do you browse Markdown-heavy repo docs?"
- "Looking for feedback on a private docs reader for local folders and GitHub repos"

Avoid posting the same pitch everywhere. Join real conversations where the product is relevant.

## X, LinkedIn, And Build In Public

Share short product clips and practical development notes.

Post ideas:

- Demo clip opening a local Markdown folder
- Demo clip browsing repo docs
- Screenshot of the file tree and rendered Markdown
- Thread: why local-first tools matter for documentation
- Thread: why hosted documentation tools are overkill for many Markdown folders

Goal:

Build founder-led visibility and make the product easy to understand at a glance.

## Dev.to And Hashnode

Publish practical articles for developers.

Article ideas:

- "Building a local-first Markdown reader with Next.js"
- "How to browse Markdown docs without uploading files"
- "Why local-first tools matter for developer documentation"

Goal:

Create long-term search traffic and developer trust.

## Direct Outreach

Reach out to maintainers of Markdown-heavy open-source repositories.

Sample message:

> Hey, I built mdocks, a local-first Markdown reader for browsing folders and repo docs. Your project has a lot of Markdown docs, so I thought it might be useful for contributors. No account or upload needed. I would love feedback if you try it.

Goal:

Get focused feedback from people with real documentation pain.

## Launch Timeline

## Week 1: Foundation

- Tighten homepage copy.
- Add screenshots or a short GIF.
- Improve the GitHub README.
- Add GitHub social preview.
- Add analytics events for demo clicks, command copy clicks, and GitHub clicks.

## Week 2: Soft Launch

- Publish two focused SEO pages.
- Share with small developer communities.
- Ask 10 developers for feedback.
- Fix onboarding confusion.

## Week 3: Public Developer Launch

- Launch on Show HN.
- Post in relevant Reddit communities.
- Share X and LinkedIn demo clips.
- Collect quotes, objections, and bug reports.

## Week 4: Product Hunt

- Launch with polished assets.
- Ask existing users for honest support.
- Follow up with a public "what we learned" post.

## Metrics To Track

Track the actions that show real interest:

- Homepage visitors
- Open demo clicks
- Run command copy clicks
- GitHub clicks
- GitHub stars
- Demo document opens
- Returning visitors
- Issues or feedback created
- Newsletter signups if added later

## Simple Funnel

Visitor lands on website.

They understand "private local Markdown reader" in five seconds.

They open the demo or copy the run command.

They star GitHub or install mdocks.

They give feedback or share it with another developer.

## Core Message To Repeat

Use this message consistently:

> Private Markdown browsing for local files and GitHub repos.

This is the clearest marketable idea for mdocks.
